## 6. 総評

### 6.1 全体の変換精度

3ステップ合計の品質評価:

| ステップ | 成功率 | 品質スコア |
|--------|------|---------|
| PDF → JSON | 91.7%（11/12） | 良好（スキップセクションは5件で発生） |
| JSON → DOCX | 100%（11/11） | 優秀（平均保持率98.2%） |
| DOCX → JSON | 100%（11/11） | 良好（平均値保持率78.8%） |

フルラウンドトリップ（3ステップ全完走）: **11/12件（91.7%）**

**フルラウンドトリップ実効保持率: 78.8%**  
（Section 5.2 の値保持率はすでに output.json → rt_NN.docx → rt2_NN.json の2ステップを通した結果であり、これがエンドツーエンドの保持率そのものである。Step 2 の98.2%はStep 3の上限を示すもので、両者を掛け合わせると二重計算になる点に注意。）

### 6.2 ステップ別の品質特性

**JSON→DOCX（Step2）**が最も高品質（98.2%）であり、DOCXへのレンダリングはほぼ完璧に動作している。セクション見出し16件はすべて正しく日本語で出力されており、英語ラベルの残存は元データが英語を含む場合のみで、変換ツール由来の問題はない。

**DOCX→JSON（Step3）**は平均78.8%の保持率で、主な情報損失は以下に起因:
- 複数行の長文テキストがDOCXからの再抽出時に截断または統合される
- StabilityReactivityセクションのスキーマ不一致（3ファイルで再現）
- 連絡先フィールド（Fax、Email）の再抽出漏れ

### 6.3 残存課題と改善提案

1. **StabilityReactivityスキーマ不一致（優先度: 高）**  
   PDF→JSON および DOCX→JSON の双方で繰り返し発生。HazardousDecompositionProductsの値が配列か文字列かをLLMが状況に応じて切り替えており、スキーマ定義側の受け入れ型を広げる（anyOf）か、LLMプロンプトで型を明示する対策が有効。

2. **複数行テキストの保持（優先度: 高）**  
   DOCX→JSON の最大の情報損失源。DOCXのレンダリング時に長文を段落分割している場合、to-jsonでの再抽出時に情報が失われる。段落結合ロジックの強化、またはプロンプトで「段落をまたいで全文を結合すること」と明示することを推奨。

3. **input01.pdf の vision OCR JSONパースエラー（優先度: 中）**  
   LLMの出力が途中で切れているためJSONが不正になっている。max_tokensを増やすか、部分的な結果をページ分割で複数回取得してマージする方式を検討すべき。

4. **言語自動検出の精度（優先度: 低）**  
   vision OCRを使用したファイル（02, 09, 12）で英語と誤検出。これらは実際には日本語SDSの可能性がある。「--lang ja」を明示的に指定することで回避可能だが、自動検出ロジックの改善も検討する価値がある。

5. **連絡先情報の保持（優先度: 低）**  
   FaxやEmailアドレスが複数ファイルで失われている。LLMプロンプトで「連絡先情報を必ず抽出すること」と強調するか、専用の後処理抽出ロジックを追加することで改善できる可能性がある。
